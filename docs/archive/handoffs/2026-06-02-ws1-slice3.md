## 2) What Was Completed
- Library data contract exposed via two typed endpoints:
  - `GET /api/library/agents` (optional filters: `view`, `fn`, `industry`, `status`, `exp`, `collection`)
  - `GET /api/library/agents/[agentId]`
- Per-agent launch metadata: `launchType` (`open_url` | `guided_tour` | `unavailable`), `launchUrl`, `availabilityReason`.
- UI wired to an API-backed source: pages call the same `lib/library-service` module the API routes use (single source of truth). API routes are the external HTTP contract.
- Real, explicit card/detail actions:
  - Open Agent → opens `launchUrl` (accent button).
  - View Guided Tour → opens guided flow `launchUrl`.
  - Unavailable → disabled action + clear `availabilityReason` text.
- Server-side validation/error handling:
  - Unknown `agentId` → `404 { error: { code: "agent_not_found" } }`.
  - Malformed `agentId` (non-slug) → `400 { error: { code: "invalid_agent_id" } }`.
  - Unrecognized filter value → `400 { error: { code: "invalid_filter", details: [...] } }`.
  - UI stays resilient: `sanitizeFilters` coerces unknown query values to `all` so hand-edited URLs never break the page.
- Layout preserved: cards remain centered, filters/collections on the right rail; only the action area changed for function.
